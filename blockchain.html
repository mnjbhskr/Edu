The Mathematics of Blockchain

Building trust in a trustless world — cryptographic foundations of decentralised systems

Opening insight: Blockchains solve a fundamental problem in computer science: how can strangers trust each other without a bank, government, or company in the middle? The answer lies in mathematics. Using cryptographic hash functions, Merkle trees, proof-of-work puzzles, and digital signatures, a blockchain creates a tamper-proof ledger where the mathematics itself enforces the rules. No middleman needed.

1. Hash Functions — Digital Fingerprints

A cryptographic hash function is a mathematical operation that takes any input (a word, a number, a document, a transaction) and produces a fixed-length output—a "hash"—that uniquely identifies that input. Think of it as a fingerprint for data.

Key Properties of Cryptographic Hashes
Deterministic: The same input always produces the same output.
One-way: You cannot reverse the hash to find the original input.
Avalanche effect: Even a tiny change to the input—one character, one bit—produces a completely different hash.
Collision resistant: It is computationally infeasible to find two different inputs that produce the same hash.

In blockchain, hashes serve two purposes: (1) they compress large transactions into small fingerprints, and (2) they detect tampering. Change one transaction, and the hash changes entirely. This avalanche property is crucial.

Why This Matters for Blockchain
When a transaction is hashed, that hash becomes a compact "fingerprint." If someone tries to alter even one number in the transaction, the hash changes completely. The next block in the chain stores the hash of the previous block, so changing an old transaction breaks the chain. This creates tamper-evidence: any alteration is immediately detectable.

2. Merkle Trees — Efficient Verification

A block in a blockchain typically contains many transactions—sometimes thousands. To prove that a specific transaction is in the block, you don't need to download and verify all the other transactions. Instead, you use a Merkle tree, a clever binary tree structure where each node is the hash of its two children, and the root is called the "Merkle root."

How a Merkle Tree Works
Each leaf is the hash of a single transaction. Each parent node is the hash of the concatenation of its two children's hashes. The root combines all information about all transactions. To verify a single transaction, you only need the hashes along the path from that leaf to the root—about log₂(n) hashes for n transactions. This is far more efficient than re-hashing everything.

Merkle Proofs in Practice
Bitcoin uses Merkle trees in every block header. The Merkle root is just 32 bytes, but it summarises all transactions. A "Merkle proof" is a small set of hashes (about 20 for a block with 2,000 transactions) that proves a transaction is in the block. This is how "light clients" (like mobile wallets) can verify transactions without storing the entire blockchain.

3. Proof of Work — The Mining Puzzle

How does a blockchain prevent attackers from modifying the past? By making it expensive to rewrite history. Proof-of-work is a puzzle: miners must find a special number called a "nonce" such that the hash of the block (which includes the nonce) meets a specific condition—usually, the hash must start with a certain number of leading zeros.

The Mining Problem
Find a nonce N such that:
hash(block_data + N) starts with k leading zeros.

There is no shortcut: miners must try nonces at random until one works. The expected number of attempts is 2k. This makes mining hard, and changing a historical block means re-mining that block and all subsequent blocks.

Why Proof of Work Secures Blockchain
Once a block is mined, it is locked in place by computational work. To tamper with a transaction in an old block, an attacker must: (1) modify the transaction, (2) recalculate the Merkle root, (3) re-mine that block, and (4) re-mine all subsequent blocks. Meanwhile, the honest network is still adding new blocks. To catch up, the attacker needs more than 50% of the network's computing power. This is why blockchains with large networks are secure: the cost of rewriting history exceeds the value of the fraud.

4. Building a Chain — Tamper-Evidence in Action

Each block contains the hash of the previous block. This links blocks into a chain. If someone tampers with a transaction in block 2, the hash of block 2 changes. But block 3 stores the old hash of block 2, so block 3 is now invalid. And block 4 depends on block 3, so it is invalid too. The tampering is immediately obvious, and the attacker must re-mine blocks 2, 3, 4, and every subsequent block to cover their tracks. For a chain with hundreds of blocks, this is computationally prohibitive.

5. Digital Signatures — Proving Ownership

A blockchain is not just a tamper-proof ledger; it must also prove ownership. If person A sends Bitcoin to person B, how does the network know that A actually authorised the transaction? Through a digital signature based on elliptic-curve cryptography (specifically, ECDSA—Elliptic Curve Digital Signature Algorithm).

How Digital Signatures Work
Each user has a pair of keys: a private key (secret) and a public key (known to everyone). The private key is a very large random number. The public key is derived from the private key using elliptic-curve maths.

To sign a transaction, you compute a signature using your private key and the transaction data. Anyone can verify the signature using your public key, but only you can create a valid signature. This proves you authorised the transaction without revealing your private key.

The Elliptic Curve secp256k1

Bitcoin uses the elliptic curve secp256k1, defined by the equation:

y² ≡ x³ + 7 (mod p)

where p = 2256 − 232 − 977. This curve has a special structure that makes cryptographic operations efficient. The public key is a point on this curve; the private key is a scalar (a number) that generates the public key via repeated point multiplication.

Signature Verification on the Blockchain

When a transaction is broadcast to the network, it includes:

• Transaction data (sender, receiver, amount)
• Signature (created using the sender's private key)
• Public key (or its hash, known as the Bitcoin address)

Any node can verify the signature using the public key. If the signature is valid, the transaction is authentic. If someone tries to forge a signature or alter the transaction, the verification fails. This is mathematically infeasible to forge because no one knows the private key.

Linking Digital Signatures to Hash Functions
The signature is computed over the hash of the transaction, not the transaction itself. This means: (1) the signature is compact (the hash is fixed-length), and (2) any change to the transaction invalidates the signature. Combined with the chain-linking property, this creates a system where each transaction is locked in place by mathematics.

Why secp256k1?

Bitcoin chose secp256k1 because it is (1) efficient to implement, (2) well-studied by cryptographers, (3) resistant to side-channel attacks, and (4) produces 256-bit security (2256 possible private keys, which is astronomically large). Modern blockchains sometimes use other curves, such as Curve25519 (Ed25519 signatures), but the principle remains the same: elliptic-curve cryptography provides a mathematical proof of ownership.

Putting It Together: The Complete Picture

A blockchain combines four mathematical ideas into one cohesive system:

1. Hash Functions
Compress data into fixed-length fingerprints. Any change to data changes the hash entirely.
2. Merkle Trees
Summarise many transactions with a single root hash. Allow efficient verification of any single transaction.
3. Proof of Work
Make rewriting the past exponentially harder by requiring miners to solve a computational puzzle for each block.
4. Digital Signatures
Prove ownership and authenticity without revealing secrets. Based on elliptic-curve cryptography.

Together, these tools create a system where:

  • Immutability: Changing an old transaction breaks the chain.
  • Transparency: Everyone can verify the ledger independently.
  • Decentralisation: No central authority is needed to enforce the rules; the mathematics does.
  • Security: Attacking the system costs more than the value of fraud.
The deeper insight: A blockchain is not just a database; it is a mathematical system for creating trust. By combining one-way functions, tree structures, computational puzzles, and public-key cryptography, we can build a ledger that is transparent, immutable, and decentralised. No bank, government, or company is needed. The mathematics itself enforces the rules. This is the power of blockchain.

Further Exploration

To deepen your understanding, explore these related topics:

  • Elliptic Curve Cryptography: The mathematical foundation of digital signatures. Related to other asymmetric encryption systems used on the internet.
  • Proof of Stake: An alternative to proof-of-work that uses far less energy. Based on game theory and economic incentives rather than computational puzzles.
  • Zero-Knowledge Proofs: Advanced cryptographic techniques that allow you to prove a statement is true without revealing any information about it.
  • Lattice-Based Cryptography: A potential post-quantum alternative to elliptic-curve methods, resistant to attacks from quantum computers.
