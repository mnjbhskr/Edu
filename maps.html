The Mathematics of Maps

How routing algorithms shape the digital world — a journey through graphs, distances, and optimal paths

Every time you ask Google Maps for directions, or your satnav recalculates a route, a graph algorithm runs in milliseconds on a network of millions of nodes and billions of edges. Behind that smooth blue line connecting you to your destination lies centuries of mathematical elegance: the discovery that the problem of finding the shortest path can be solved systematically, and that even impossible-seeming problems can be approximated with surprising accuracy.

Section 1: Graphs — The Language of Maps

A map is fundamentally a graph: intersections are nodes (or vertices), roads are edges, and the distance or travel time is the weight of each edge.

Understanding Graph Structure

In the interactive canvas below, each coloured circle is a node representing a city intersection. Each line connecting two nodes is an edge. The number on each edge is its weight — in the real world, this could be kilometres or minutes of travel time.

Why brute force fails: With just 10 cities, there are 10! = 3.6 million possible routes. With 20 cities, that's 20! ≈ 2.4 quintillion. We need smarter algorithms.

Section 2: Dijkstra's Algorithm — Guaranteed Shortest Path

In 1956, Dutch computer scientist Edsger Dijkstra was sitting in an Amsterdam café when he conceived an elegant algorithm that would become fundamental to routing systems worldwide. Given a starting point, Dijkstra's algorithm finds the shortest path to every other node in the graph.

How Dijkstra's Works

  1. Mark all nodes unvisited. Set the start node's distance to 0, all others to infinity.
  2. From the current node, examine all unvisited neighbours. Update their tentative distances if we found a shorter path.
  3. Mark the current node as visited (coloured green).
  4. Select the unvisited node with the smallest tentative distance as the next current node.
  5. Repeat until all nodes are visited or the destination is reached.
Complexity: O((V + E) log V) with a min-heap
V: Number of vertices (nodes)
E: Number of edges

Section 3: A* — Dijkstra with a Compass

Dijkstra's algorithm explores all directions equally, like searching with no destination in mind. A* (A-star) uses a heuristic — an educated guess about the direction to the goal — to explore fewer nodes and find the path faster.

The A* Heuristic

Instead of just tracking the actual cost from the start (g(n)), A* also estimates the remaining distance to the goal (h(n)). It prioritises nodes where f(n) = g(n) + h(n) is smallest.

f(n) = g(n) + h(n)
g(n) = actual distance from start   |   h(n) = estimated distance to goal
Key advantage: A* can find the optimal path with fewer node explorations than Dijkstra, often 5–10× faster on large graphs.

Section 4: The Travelling Salesman Problem

Imagine a salesman who must visit every city on a map exactly once and return home. Which route minimises total distance? This is the Travelling Salesman Problem (TSP), one of the most famous problems in computer science and mathematics.

Why TSP Matters

The TSP appears everywhere: in logistics, circuit board manufacturing, DNA sequencing, and telescope scheduling. It's easy to understand but computationally brutal. With n cities, there are (n-1)!/2 possible routes. With just 25 cities, that's 1.5 × 10²³ possibilities — more than the number of stars in the observable universe.

NP-Hard Problem: No known algorithm solves TSP in polynomial time. The Clay Mathematics Institute offers a $1 million prize for proving whether P = NP. If P = NP, TSP becomes tractable. If not, some problems are fundamentally hard.

Section 5: How Google Maps Actually Works

Dijkstra's and A* are beautiful, but they're too slow for routing across continental-scale road networks with tens of millions of nodes. Real mapping systems use far more sophisticated algorithms.

Contraction Hierarchies (CH)

This algorithm pre-processes the graph to create a hierarchical structure. The most important nodes (like motorway junctions) are identified and used to shortcut queries. Instead of exploring millions of nodes, a route query might examine only thousands. Query time drops from seconds to milliseconds.

ALT Algorithm (A*, Landmarks, Triangle Inequality)

A* needs accurate heuristics. ALT pre-computes distances to a small set of "landmark" nodes spread across the map. Using the triangle inequality, it can bound the remaining distance tightly, guiding search more effectively than Euclidean distance alone.

Real-Time Traffic and Rerouting

When live traffic is incorporated, weights on edges change dynamically. Modern systems constantly recompute key routes and alert drivers to congestion. This turns a static pathfinding problem into a dynamic, real-time system. Some edges are recomputed every few seconds.

Multi-Modal Routing

Modern navigation handles not just cars, but public transport, cycling, walking, and combinations (e.g., drive to station, take train, walk to destination). This requires graphs that model different vehicle types and transfer points, exponentially increasing complexity.

The bottom line: Google Maps, Apple Maps, and similar services combine multiple algorithmic layers: preprocessing for speed, live traffic integration for accuracy, and multi-modal graphs for flexibility. The mathematics behind the scenes is remarkably elegant.
