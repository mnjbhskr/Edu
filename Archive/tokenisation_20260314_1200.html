Tokenisation

Converting text and sequences into tokens

Before a language model can read your words, it must break them into pieces called tokens. A token might be a whole word, a fragment like "un" or "ing", or even a single character. The choice of tokenisation method profoundly affects what the model can learn, how efficiently it processes text, and how it handles languages it has never seen. Let us explore three approaches and see the trade-offs first-hand.

1. What Are Tokens?

Key insight: Character-level tokenisation produces many tiny tokens (the model must learn spelling from scratch). Word-level creates fewer tokens but cannot handle unseen words. BPE strikes a balance — common words stay whole, rare words are broken into meaningful subpieces.

How BPE works: Start with individual characters. Count every adjacent pair in the corpus. Merge the most frequent pair into a single new token. Repeat. Common subwords like "low" emerge naturally from the statistics of the training data.

3. Tokenisation Quirks

Surprising behaviours in real tokenisers

Numbers

"123456789"
["123", "456", "789"]
Large numbers are often split into 3-digit chunks. The model does not "see" the full number — it sees pieces and must learn arithmetic from fragments.

Leading Spaces

" hello" vs "hello"
[" hello"] vs ["hello"]
Many tokenisers treat the space before a word as part of the token. " hello" and "hello" become different tokens entirely, changing how the model interprets them.

Non-English Text

"hello" → 1 token
"こんにちは" → 5+ tokens
Tokenisers trained mostly on English use more tokens for other languages. Japanese, Arabic, and Hindi text may need 3–5× more tokens than equivalent English, making processing slower and costlier.

Code

"def foo():"
["def", " foo", "()", ":"]
Programming syntax creates unusual token boundaries. Indentation (spaces/tabs) may consume many tokens. Common patterns like "()" often merge into single tokens through BPE training.
These quirks are not bugs — they are consequences of statistical learning. A tokeniser optimises for the most common patterns in its training data, which means less common patterns (rare languages, large numbers, unusual formatting) pay a cost in token count.

4. Vocabulary Size Trade-offs

The balancing act between vocabulary size and efficiency

5,000
The sweet spot: A small vocabulary means fewer parameters in the embedding table but longer token sequences (more compute per sentence). A large vocabulary means shorter sequences but a massive embedding table. Modern LLMs typically use 30k–100k tokens — a carefully chosen middle ground.

5. What Have We Learnt?

Summary

Tokens are the atoms of language models. Every piece of text — whether a novel, a tweet, or a line of code — must first be split into tokens before a model can process it.
BPE learns a vocabulary from data. Starting from individual characters, it repeatedly merges the most frequent pair. This produces a vocabulary where common words are single tokens and rare words are broken into reusable subpieces.
Vocabulary size is a design choice with real consequences. Smaller vocabularies produce longer sequences; larger vocabularies need more memory for embeddings. The right balance depends on the model’s goals and the diversity of its training data.
Tokenisation shapes what a model can see. If a number is split across tokens, the model must learn arithmetic from fragments. If a language uses more tokens per sentence, it is effectively "more expensive" for the model to process.
Where next? Tokenisation is the first step in a pipeline. Once text is tokenised, each token is mapped to a vector via an embedding table (see: Embeddings). The model then predicts the next token in the sequence (see: Next-Token Prediction), which is the fundamental operation behind modern language models.
