How Passwords Are Stored

The cryptography, mathematics, and paradoxes that protect (and fail to protect) your secrets — Advanced

In June 2012, LinkedIn suffered a catastrophic breach: 6.5 million user passwords leaked online. But these weren't plaintext passwords—they were hashes, computed with the supposedly secure SHA-1 algorithm. The problem: they were stored without salting, all in one column, making them vulnerable to rainbow table attacks. Within hours, security researchers had cracked millions of them. This disaster, and many others since, reveal a fundamental truth: storing passwords demands more than a hash function. It requires understanding hashing, salting, and the surprising mathematics of the birthday paradox. This page explores the cryptography and mathematics that defend your passwords—and the clever attacks that circumvent them.

Section 1: Why Not Store Plaintext?

A Brief History of Catastrophe

For decades, many systems stored passwords in plaintext: the user types "mysecret", and the server saves "mysecret" in the database. This is disastrous. If a hacker breaches the database, they instantly have every user's password. And users don't just reuse one password—they use the same password across Gmail, Twitter, Facebook, their bank. One breach cascades into a dozen compromised accounts.

The solution: never store the password itself. Instead, store a hash of the password. A hash function transforms input into a fixed-length output that is (practically) impossible to reverse. When the user logs in, hash their input and compare the result to the stored hash. If they match, the password is correct.

Section 2: Hash Functions for Passwords

Why SHA-256 Is Not Enough

Not all hash functions are created equal. A "fast" hash function like SHA-256 is perfect for checksums and data integrity checks. But for passwords, speed is a vulnerability. If a hash function runs in microseconds, an attacker can try billions of passwords per second, making brute-force attacks feasible.

Password hashes must be slow by design. Algorithms like bcrypt, scrypt, and Argon2 deliberately consume CPU and memory, making each attempt take milliseconds or longer. This throttles attackers while imposing only a minor cost on legitimate users.

Properties of a Good Password Hash Function

Deterministic
Hash the same password a hundred times and get the same result every time. This is essential for verification.
One-Way
Practically impossible to reverse: given a hash, you cannot efficiently recover the original password.
Avalanche Effect
A tiny change in input produces a completely different output. This destroys patterns and makes educated guessing futile.
Slow
Takes hundreds of milliseconds per hash (not microseconds). Throttles brute-force attacks, adds negligible delay to login.

Section 3: Rainbow Tables and Salting

Pre-Computed Attacks

An attacker doesn't need to hash passwords in real-time. They can pre-compute a massive lookup table: password → hash. This is a rainbow table. Given a stolen hash, a lookup in this table instantly reveals the password—no brute-force required.

For common passwords, rainbow tables are publicly available online. Millions of passwords have been pre-computed and indexed.

The Solution: Salting

The defence is elegantly simple: add a random salt to each password before hashing. If the user's password is "password123" and the salt is "xK9m", you hash "password123xK9m" instead. Now, even though millions of users might share the password "password123", each one has a different salt, producing different hashes. The rainbow table becomes useless.

Section 4: The Birthday Paradox

A Counter-Intuitive Probability

In a classroom of 23 random people, what is the probability that two share a birthday? Most people guess around 10–20%. The true answer: approximately 50%. This is the famous birthday paradox, and it has profound implications for cryptography.

The paradox stems from a combinatorial fact: we're not computing the probability that someone shares a birthday with a specific person. We're computing the probability of any pair matching. With 23 people, there are C(23, 2) = 253 pairs, and each has a 1/365 chance of matching. The probabilities compound.

Generalised: given a set of n items drawn randomly from a space of N possible values, collisions become likely when n ≈ √N.

Section 5: Birthday Attacks on Hash Functions

Collisions in Cryptographic Hashes

If a hash function produces n-bit outputs, there are 2^n possible hash values. By the birthday paradox, we expect collisions (two different inputs with the same hash) to occur after approximately 2^(n/2) trials.

This is a fundamental limit. A 128-bit hash (like MD5) can be attacked in ~2^64 operations, not 2^128. This makes 128-bit hashes far weaker than they initially appear. A 256-bit hash (like SHA-256) requires ~2^128 operations to find a collision—effectively unbreakable with current technology.

This is why modern cryptographic systems use at least 256-bit hashes and why SHA-1 (160 bits, vulnerable to ~2^80 attacks) has been deprecated.

Historical Impact: The SHA-1 Collision

In 2017, Google announced they had found the first practical collision in SHA-1, called "SHAttered". They computed two distinct PDF files with the same SHA-1 hash, taking ~6,500 GPU years of compute. This was theoretically inevitable by the birthday paradox but practically shocking. It proved that SHA-1, used since 1995, was finally broken and should be abandoned.

Section 6: Password Entropy and Strength

Measuring Password Strength

Password entropy quantifies the "search space" an attacker must explore. If a password is chosen from C possible character combinations and has length L, there are C^L possible passwords. Entropy is log₂(C^L) = L × log₂(C) bits.

A password with 50 bits of entropy is roughly as strong as a 50-bit key: guessing requires about 2^50 attempts on average. At 100 billion guesses per second, this takes 355 million seconds ≈ 11 years. At 1 million guesses per second (realistic with bcrypt), it takes 355 million seconds = 11 years. With a slow hash, you're safe.

Character Sets and Entropy

Digits Only
Character set: 10. A 6-digit PIN = log₂(10^6) ≈ 20 bits. Weak.
Lowercase + Digits
Character set: 36. An 8-character password = log₂(36^8) ≈ 41 bits. Fair.
Mixed Case + Digits + Symbols
Character set: 94. An 8-character password = log₂(94^8) ≈ 52 bits. Good.
Passphrase (4 Common Words)
Dictionary: ~10,000 words. 4 words = log₂(10,000^4) ≈ 53 bits. Good, easier to remember.

Real-World Examples

The Paradox of Passwords: Complex symbols and numbers feel strong but only add a small constant factor. "correct horse battery staple" (4 common words) has ~52 bits of entropy and is far easier to remember than "P@ssw0rd!" (also ~52 bits). Passphrases win.
