Gradient Descent

The algorithm that learns — how a model navigates a mountain range of errors to find the valley of understanding.

The Learning Problem

Every neural network begins as pure randomness. Its billions of parameters are initialised with small random values. Asked to predict anything, it produces nonsense. Learning is the process of adjusting these parameters until the predictions become good.

But "adjusting parameters" is deceptively simple language for a profound problem. A modern language model has 70 billion parameters. The space of possible configurations is unimaginably vast. How do you search it efficiently? The answer — elegant in its simplicity — is to follow the gradient.

The Key Insight

At any point in parameter space, the gradient of the loss function tells you the direction of steepest ascent — the direction in which error increases fastest. Step in the opposite direction — downhill — and you reduce the error. Repeat millions of times. This is gradient descent.

The Loss Function

Before descending, you need a landscape to descend. The loss function L(theta) measures how wrong the model currently is, given parameters theta. It maps every possible configuration of parameters to a single number: the error.

For a model with two parameters (a toy case), the loss function is a surface — a literal landscape you can visualise. Valleys are good configurations; peaks and plateaus are bad. The goal: find the deepest valley.

Loss function L(theta): R^n -> R (n parameters -> one error number) Common choices: Mean Squared Error: L = (1/N) Sum_i (y_i - y_hat_i)^2 -- regression Cross-Entropy: L = -(1/N) Sum_i y_i log(y_hat_i) -- classification KL Divergence: L = Sum_i p(i) log(p(i)/q(i)) -- distributions The gradient: nabla L(theta) = [dL/dtheta_1, dL/dtheta_2, ..., dL/dtheta_n] A vector pointing in the direction of steepest ascent.

What To Notice
  • Too small eta: the ball inches downhill painfully slowly. Training would take forever.
  • Too large eta: the ball bounces over the valley and may diverge entirely. Loss goes up instead of down.
  • Different starting positions can lead to different local minima — the landscape has multiple valleys.
Why It Matters
  • The learning rate is the single most important hyperparameter in training. Get it wrong and nothing works.
  • Real training uses learning rate schedules that change eta over time — warm up, then decay.
  • The existence of multiple local minima is why training neural networks is hard but also why different runs produce different (but equally good) models.

The Update Rule

At each step, the algorithm computes the gradient at the current position and takes a small step in the opposite direction. The size of the step is controlled by the learning rate eta — arguably the most important hyperparameter in deep learning.

Gradient descent update: theta <- theta - eta * nabla L(theta) theta : current parameters (position in the landscape) eta : learning rate (step size) nabla L: gradient (direction of steepest ascent) -nabla L: direction of steepest descent -- we move here Too large eta -> overshoot, bounce around, diverge Too small eta -> progress glacially slow, get stuck
The learning rate problem

If eta is too large, the ball bounces over valleys and diverges. If eta is too small, training takes impractically long. Finding a good eta is partly art, partly science. Modern practice uses adaptive learning rate schedules: warm-up from a small value, hold at a peak, then decay. GPT-3 used a cosine decay schedule over 300 billion tokens.

Stochastic Gradient Descent

With billions of training examples, computing the true gradient — which requires looking at every example — is prohibitively expensive. Stochastic Gradient Descent (SGD) uses a random mini-batch of, say, 32 examples to estimate the gradient. The estimate is noisy, but each step is computed 10 million times faster.

Counterintuitively, this noise helps: it prevents the optimizer from settling too confidently in a poor local minimum. The stochastic jitter acts as a form of exploration.

Try This
  • Run with noise = 0. Both paths are identical — pure gradient descent.
  • Increase noise to 0.40. SGD wobbles but often finds a better minimum.
  • Push noise to 1.0. Too much noise — SGD jumps erratically and may never settle.
What To Notice
  • Full GD follows a smooth path straight to the nearest local minimum — even if it is not the best one.
  • SGD's noisy path sometimes escapes shallow local minima and finds deeper valleys.
  • There is an optimal noise level: enough to explore, not so much that you cannot converge.
Why It Matters
  • Real training always uses mini-batch SGD — computing the exact gradient over all data is too expensive.
  • The noise in SGD acts as implicit regularisation, helping the model generalise better.
  • Batch size (which controls noise level) is a key hyperparameter alongside learning rate.
SGD
theta <- theta - eta*nabla L
Vanilla. Fast per step. Sensitive to learning rate. Used in initial deep learning breakthroughs (AlexNet, 2012).
SGD + Momentum
v <- beta*v + nabla L
theta <- theta - eta*v
Accumulates velocity in consistent directions. Rolls through small local minima. beta typically 0.9.

Default choice for most modern models. Combines momentum with per-parameter rate adaptation. Kingma & Ba, 2014.

Why Neural Networks Are Hard to Optimise

The loss landscape of a deep neural network is not a smooth bowl with one valley. It is a high-dimensional surface with saddle points (neither min nor max), flat regions (vanishing gradients), sharp cliffs (exploding gradients), and millions of local minima.

Modern deep learning works despite this complexity for reasons not fully understood. One partial explanation: with enough parameters, most local minima are nearly as good as the global minimum, because the network is massively overparameterised — it has far more parameters than training examples, and any configuration that fits the data well is likely to generalise.

The remarkable fact

Despite this complexity, gradient descent reliably trains neural networks with hundreds of billions of parameters. The theoretical understanding lags far behind the empirical success. We know it works. We are still building the mathematics to explain precisely why it works as well as it does.

Practical training loop (simplified): for epoch in range(num_epochs): for batch in dataloader: # mini-batch of ~32-4096 examples predictions = model(batch.inputs) # forward pass loss = criterion(predictions, batch.targets) loss.backward() # compute nabla L via backpropagation optimizer.step() # theta <- theta - eta*nabla L optimizer.zero_grad() # clear accumulated gradients GPT-3: ~300 billion token updates. ~3.14x10^23 floating-point operations.
